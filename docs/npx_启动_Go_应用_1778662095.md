# npx 启动 Go 应用

#### ChatGPT:
我会查一下这个包在 npm/GitHub 上的实际结构，重点看它的 `package.json`、`bin` 字段和安装脚本。

因为 **Bifrost 不是让 `npx` 直接运行 Go 源码**，而是发布了一个 **npm 包 `@maximhq/bifrost`**，这个 npm 包里放了一个很薄的 Node.js 启动脚本，脚本再去下载并执行真正的 Go 二进制。

整体链路是：

```txt
npx -y @maximhq/bifrost
        │
        ▼
npm 下载 @maximhq/bifrost 这个 npm 包
        │
        ▼
读取 package.json 的 bin
        │
        ▼
执行 bin.js
        │
        ▼
bin.js 判断系统/CPU 架构
        │
        ▼
下载对应平台的 bifrost-http Go 二进制
        │
        ▼
启动 bifrost-http
```

## 关键点 1：它是 npm 包

npm 页面上可以看到 `@maximhq/bifrost` 是一个 public npm package，并且仓库地址指向 `github.com/maximhq/bifrost`。npm 页面还显示这个包体积很小，只有 2 个文件、0 个依赖，这也符合“npm 包只是启动器/包装器”的模式。([NPM](https://www.npmjs.com/package/%40maximhq/bifrost))

Bifrost 的 README 也明确把启动方式写成：

```bash
npx -y @maximhq/bifrost
```

并且仓库结构里专门有一个 `npx/` 目录，说明它就是为了 NPX 一键安装/启动准备的。([GitHub](https://github.com/maximhq/bifrost))

## 关键点 2：`package.json` 里配置了 `bin`

npm 的规则是：一个包如果在 `package.json` 里声明了 `bin` 字段，npm/npx 就能把它当成命令执行。npm 文档说明，`bin` 是“命令名 → 本地文件”的映射，安装后该文件会被链接成可执行命令。([npm 文档](https://docs.npmjs.com/cli/v10/configuring-npm/package-json))

Bifrost 的 `npx/package.json` 里就是这样配置的：

```json
{
  "name": "@maximhq/bifrost",
  "bin": {
    "bifrost": "bin.js"
  },
  "type": "module",
  "dependencies": {}
}
```

对应源码快照里能看到 `bin` 字段把命令 `bifrost` 指向 `bin.js`。([GitHub](https://github.com/vsrc/maximhq-bifrost/blob/main/npx/package.json))

所以当你执行：

```bash
npx -y @maximhq/bifrost
```

npm 会发现这个包有一个可执行入口 `bifrost -> bin.js`，于是执行它。npm 文档也说明，如果包只有一个 `bin`，或者 bin 名称能匹配包名的非 scope 部分，`npx` 会自动推断要运行哪个命令；`-y` / `--yes` 则是跳过安装确认提示。([npm 文档](https://docs.npmjs.com/cli/v10/commands/npx))

## 关键点 3：真正跑的是 Go 二进制，不是 Node 服务

`bin.js` 开头是：

```js
#!/usr/bin/env node
```

这说明 `npx` 首先执行的是 Node 脚本。脚本里定义了下载地址：

```js
const BASE_URL = "https://downloads.getmaxim.ai";
```

然后解析 `--transport-version` 参数，默认使用 `latest`。([GitHub](https://github.com/vsrc/maximhq-bifrost/blob/main/npx/bin.js))

接着它会根据当前机器判断平台和架构：

```txt
darwin / linux / windows
arm64 / amd64 / 386 ...
```

并选择二进制名：

```txt
bifrost-http
bifrost-http.exe
```

相关逻辑在 `getPlatformArchAndBinary()` 里。([GitHub](https://github.com/vsrc/maximhq-bifrost/blob/main/npx/bin.js))

最后它拼出下载 URL：

```js
`${BASE_URL}/bifrost/${VERSION}/${platformDir}/${archDir}/${binaryName}`
```

下载到临时目录，`chmod 755`，然后用 `execFileSync` 执行这个二进制。([GitHub](https://github.com/vsrc/maximhq-bifrost/blob/main/npx/bin.js))

所以本质上是：

```txt
npx 运行 Node wrapper
Node wrapper 下载 Go 编译产物
Node wrapper exec Go 程序
```

## 你自己的 Go 项目想仿照它，可以这样做

你的 npm 包结构可以是：

```txt
npx/
  package.json
  bin.js
```

`package.json`：

```json
{
  "name": "@your-scope/your-go-app",
  "version": "0.1.0",
  "type": "module",
  "bin": {
    "your-go-app": "bin.js"
  },
  "engines": {
    "node": ">=18.0.0"
  },
  "dependencies": {}
}
```

`bin.js` 负责：

```txt
1. 判断 process.platform / process.arch
2. 拼出下载地址
3. 下载对应平台的 Go 二进制
4. chmod +x
5. child_process.execFileSync 启动它
```

用户就可以这样启动：

```bash
npx -y @your-scope/your-go-app
```

一句话总结：**Bifrost 能用 `npx -y @maximhq/bifrost`，不是因为 npx 会运行 Go 项目，而是因为它把 Go 应用外面套了一层 npm CLI 包；npm 包里的 `bin.js` 再下载并启动真正的 Go 可执行文件。**
